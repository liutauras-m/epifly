# Conusai Platform — Dokploy Deployment

Production deployment for **Epifly** at
`https://epifly.beta.test.cloud.conusai.com` with supporting services at
`*.epifly.beta.test.cloud.conusai.com`, using [Dokploy](https://dokploy.com)
+ Traefik on the shared `dokploy-network`.

## Architecture

Each folder below is registered as an independent **Compose** application
inside the same Dokploy **Project**. Splitting them means infra never
restarts when only the gateway or web is redeployed, and Dokploy can show
per-service health.

| Folder | Dokploy app name | Purpose | Public hostname |
|---|---|---|---|
| [infra/](infra/) | `conusai-infra` | Postgres, Redis, Qdrant, RustFS, Zitadel, Lago | `auth.`, `billing.`, `s3.`, `s3-console.` |
| [gateway/](gateway/) | `conusai-gateway` | Rust agent-gateway (HTTP + WS API) | `api.` |
| [capabilities/](capabilities/) | `conusai-capabilities` | Self-registering MCP services (current-time, …) | internal only |
| [web/](web/) | `conusai-web` | SvelteKit web app — also the remote target for Tauri shells | bare apex |
| [observability/](observability/) | `conusai-observability` | Jaeger + OTel collector | `traces.` |

Hostnames (with `APP_DOMAIN=epifly.beta.test.cloud.conusai.com`):

| Service | URL |
|---|---|
| Web (Epifly) | `https://epifly.beta.test.cloud.conusai.com` |
| Agent gateway | `https://api.epifly.beta.test.cloud.conusai.com` |
| Zitadel | `https://auth.epifly.beta.test.cloud.conusai.com` |
| Lago billing | `https://billing.epifly.beta.test.cloud.conusai.com` |
| RustFS S3 API | `https://s3.epifly.beta.test.cloud.conusai.com` |
| RustFS console | `https://s3-console.epifly.beta.test.cloud.conusai.com` |
| Jaeger | `https://traces.epifly.beta.test.cloud.conusai.com` |

All services share the cookie domain `.epifly.beta.test.cloud.conusai.com`
so a single Zitadel SSO session covers the web app + gateway + Lago portal.

> **Tauri desktop / iOS / Android shells** are *not* deployed via Dokploy.
> They are native binaries built in CI and distributed via the App Store,
> Play Store, or GitHub Releases. They point their `frontendDist` /
> `devUrl` at `https://epifly.beta.test.cloud.conusai.com`, which is the
> `conusai-web` service in this folder.

## Networking

All services join the external Docker network **`dokploy-network`**
(created automatically by Dokploy). Cross-stack communication uses the
fixed `hostname:` set on each service — for example the gateway reaches
Postgres at `conusai-postgres:5432` regardless of which compose project
they live in.

Public traffic is terminated by **Traefik** on `:443`
(`entrypoints=websecure`, `certresolver=letsencrypt`). Internal-only
services (Postgres, Redis, Qdrant, RustFS S3 API, current-time) have **no
Traefik labels and no host ports** — they're reachable only inside
`dokploy-network`.

## Volumes

Persistent volumes are declared `external: true` so their **lifecycle is
decoupled from the compose stack** — they survive project deletes,
accidental `docker compose down -v`, and Dokploy redeploys. This is the
production-ready setup; the trade-off is that you must pre-create the
volumes **once** on the Dokploy host before the first `conusai-infra`
deploy.

Named volumes used:

| Volume | Owner stack | Contents |
|---|---|---|
| `conusai_postgres_data` | infra | Postgres data (Zitadel + Lago + future app DBs) |
| `conusai_redis_data` | infra | Redis AOF (Lago Sidekiq queues) |
| `conusai_qdrant_data` | infra | Qdrant vector storage |
| `conusai_rustfs_data` | infra | RustFS S3 object storage |
| `conusai_redb_data` | gateway | Agent gateway embedded KV (sessions, cache) |

### Bootstrap (one time, on the Dokploy host)

In Dokploy → **Server → Terminal** (or SSH), run:

```sh
bash dokploy/bootstrap-volumes.sh
```

…or fetch it directly from GitHub:

```sh
curl -fsSL https://raw.githubusercontent.com/liutauras-m/epifly/main/dokploy/bootstrap-volumes.sh | bash
```

The script is idempotent — it skips volumes that already exist. Verify
with `docker volume ls | grep conusai_` (you should see all 5).

## Deploy order

1. **Create DNS**: easiest is one wildcard A record
   `*.epifly.beta.test.cloud.conusai.com → <dokploy host>` plus an apex A
   record for `epifly.beta.test.cloud.conusai.com` itself. Traefik then
   issues one Let's Encrypt cert per hostname automatically.
2. **Create the project** in Dokploy and add Shared Environment (paste
   `.env.production` generated by `dokploy/generate-prod-env.ts`).
3. **Bootstrap volumes** on the Dokploy host (one time):
   `bash dokploy/bootstrap-volumes.sh` — see [Volumes](#volumes) above.
4. Deploy **`conusai-infra`** → wait for all healthchecks green
   (Zitadel takes ~60 s on first boot to seed its schema).
5. Visit `https://auth.epifly.beta.test.cloud.conusai.com`, complete Zitadel
   initial-admin setup, create the Epifly project + OIDC application,
   and record the issuer URL + client ID into Shared Environment as
   `ZITADEL_ISSUER` / `ZITADEL_CLIENT_ID`.
6. Visit `https://billing.epifly.beta.test.cloud.conusai.com`, complete
   Lago onboarding, generate the API key, store as `LAGO_API_KEY`.
7. Deploy **`conusai-gateway`**.
8. Deploy **`conusai-capabilities`**.
9. Deploy **`conusai-web`**.
10. (Optional) Deploy **`conusai-observability`**.

## Per-app Dokploy settings

When creating each Compose app in Dokploy, use these values:

| Dokploy app | Compose Path | Domains (HTTPS + Let's Encrypt) | Volumes tab |
|---|---|---|---|
| `conusai-infra` | `dokploy/infra/docker-compose.yml` | `auth.${APP_DOMAIN}` → `zitadel:8080` (h2c), `billing.${APP_DOMAIN}` → `lago-api:3000`, `s3.${APP_DOMAIN}` → `rustfs:9000`, `s3-console.${APP_DOMAIN}` → `rustfs:9001` | *(empty — declared in compose)* |
| `conusai-gateway` | `dokploy/gateway/docker-compose.yml` | `api.${APP_DOMAIN}` → `gateway:8080` | *(empty)* |
| `conusai-capabilities` | `dokploy/capabilities/docker-compose.yml` | *(none — internal only)* | *(empty)* |
| `conusai-web` | `dokploy/web/docker-compose.yml` | `${APP_DOMAIN}` → `web:3000` (bare apex) | *(empty)* |
| `conusai-observability` | `dokploy/observability/docker-compose.yml` | `traces.${APP_DOMAIN}` → `jaeger:16686` | *(empty)* |

Notes:

- **Routing is already in the compose files** via Traefik labels. The
  Domains tab is optional but recommended so Dokploy can show cert
  status. If the UI lacks an h2c toggle for Zitadel, omit Zitadel from
  the Domains tab — the compose label already sets the upstream scheme.
- **Build Context**: leave default (repo root) for every app.
- **Branch**: `main`.
- **Auto-deploy**: enable for `gateway` / `capabilities` / `web`; leave
  off for `infra` and `observability` (immutable images, deploy
  manually).

## Troubleshooting

### `external volume "conusai_postgres_data" not found`

You skipped the bootstrap step. Either run
`bash dokploy/bootstrap-volumes.sh` on the Dokploy host, or create the
volume manually:

```sh
docker volume create conusai_postgres_data
# repeat for redis_data, qdrant_data, rustfs_data, redb_data
```

Then click **Redeploy** in Dokploy.

### Zitadel returns 502 / gRPC errors

Traefik must speak **h2c** to Zitadel. The compose file already sets
`traefik.http.services.zitadel.loadbalancer.server.scheme=h2c` — if
you added Zitadel via the Dokploy Domains UI as well, make sure it
either supports h2c or remove it from the UI and rely on the label.

### `web` shows CSRF errors when submitting forms

Verify `ORIGIN`, `PROTOCOL_HEADER`, `HOST_HEADER` reach the container
and that `APP_DOMAIN` in Shared Environment matches the hostname the
browser uses. SvelteKit's adapter-node refuses requests whose `Origin`
header doesn't match `ORIGIN`.

### Lago portal redirect loops

Confirm `COOKIE_DOMAIN=.epifly.beta.test.cloud.conusai.com` (with the
leading dot) is set in Shared Environment so Zitadel's SSO cookie is
visible to `billing.epifly.…`.

## Updating

Each Dokploy app has its own Git source / build trigger. Recommended:

- **infra** → manual deploy only (immutable image tags).
- **gateway / capabilities / web** → auto-deploy on `main` push.

Traefik picks up the new container labels on rolling restart with zero
downtime as long as healthchecks pass.
